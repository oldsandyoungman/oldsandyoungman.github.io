MySQL

P112

把磁盘的页数据，缓存到内存中的Buffer Pool(数据库缓冲池)，来减少磁盘IO

  • 缓冲池和查询缓存不一样：一个是原始数据，一个是查询结果

多线程下，buffer pool 是要加锁的，在 buffer pool 很大，且并发多线程的时候，可以拆分 buffer pool

  • innodb_buffer_pool_size：总大小
  • innodb_buffer_pool_instances：分成多少份

注意：超过1G，InnoDB才会分，毕竟管理也有开销

脏页的处理，如果中途宕机，需要日志来回滚

P113

存储引擎：表的类型（表处理器）

  • InnoDB的存储格式：.idb + .frm(8.0被移除)
  • MyISAM的存储格式：.sdi + .MYD + .MYI

InnoDB 和 MyISAM 的区别：

  • 是否支持外键
    • InnoDB：支持
    • MyISAM：不支持
  • 是否支持事务
    • InnoDB：支持
    • MyISAM：不支持（因此不能回滚，崩溃无法安全恢复）
  • 效率
    • InnoDB：适用大量更新、删除操作
    • MyISAM：适用大量增加、查询操作（InnoDB写的处理效率低）
  • 适用场景
    • InnoDB：适用大量数据，高并发（行锁，效率更高）
    • MyISAM：适用少量数据，低并发
  • 缓存
    • InnoDB：数据和索引都缓存
    • MyISAM：只缓存索引
  • 占用内存：
    • InnoDB：占用内存较多（索引和数据一起存的.idb ）
    • MyISAM：占用内存少（因为分别存储数据和索引( .MYD + .MYI)，所以能只加载索引）
  • COUNT(*)的执行效率
    • InnoDB：效率低，一个一个统计
    • MyISAM：效率高，有额外变量存储

P115

索引的目的：减少磁盘IO（IO的复杂度降低，效果远大于内存算法方面）

索引的优缺点

优点

  • 类似大学图书馆建书目索引，提高数据检索的效率，降低 数据库的IO成本 ，这也是创建索引最主 要的原因。

  • 通过创建唯一索引，可以保证数据库表中每一行 数据的唯一性 。

  • 在实现数据的 参考完整性方面，可以 加速表和表之间的连接 。换句话说，对于有依赖关系的子表和父表联 合查询时， 可以提高查询速度。

  • 在使用分组和排序子句进行数据查询时，可以显著 减少查询中分组和排序的时间 ，降低了CPU的消耗。

缺点

  • 创建索引和维护索引要 耗费时间 ，并 且随着数据量的增加，所耗费的时间也会增加。

  • 索引需要占 磁盘空间 ，除了数据表占数据空间之 外，每一个索引还要占一定的物理空间， 存储在磁盘上 ， 如果有大量的索引，索引文件就可能比数据文 件更快达到最大文件尺寸。

  • 虽然索引大大提高了查询速度，同时却会 降低更新表的速度 。当对表 中的数据进行增加、删除和修改的时 候，索引也要动态地维护，这样就降低了数据的维护速度。 因此，选择使用索引时，需要综合考虑索引的优点和 缺点。

P117

B+树一般不超过4层的原因：

  • 减少IO次数（IO次数就是B+树的层数）
  • 如果一个数据页16KB，我们假设能放100条数据，一个目录页能放1000条数据，那么4层B+树，已经能存1000亿的数据，大部分场景都能适用

P118

聚簇索引并不是一种单独的索引类型，而是一种数据存储方式 (所有的用户记录都存储在了叶子结点)，也就是所谓的 索引即数据，数据即索引。

术语”聚簇”表示当前数据行和相邻的键值聚簇的存储在一起

特点：

  • 使用记录主键值的大小进行记录和页的排序，这包括三个方面的含义：

    • 页内 的记录是按照主键的大小顺序排成一个 单向链表 。
    • 各个存放 用户记录的页 也是根据页中用户记录的主键大小顺序排成一个 双向链表 。
    • 存放 目录项记录的页 分为不同的层次，在同一层次中的页也是根据页中目录项记录的主键大小顺序排成一个 双向链表 。

  • B+树的 叶子节点 存储的是完整的用户记录。

    所谓完整的用户记录，就是指这个记录中存储了所有列的值（包括隐藏列）。

我们把具有这两种特性的B+树称为聚簇索引，所有完整的用户记录都存放在这个聚簇索引的叶子节点处。这种聚簇索引并不需要我们在MySQL语句中显式的使用INDEX 语句去创建， InnDB 存储引擎会 自动 的为我们创建聚簇索引。

优点：

  • 数据访问更快 ，因为聚簇索引将索引和数据保存在同一个B+树中，因此从聚簇索引中获取数据比非聚簇索引更快
  • 聚簇索引对于主键的 排序查找 和 范围查找 速度非常快
  • 按照聚簇索引排列顺序，查询显示一定范围数据的时候，由于数据都是紧密相连，数据库不用从多 个数据块中提取数据，所以 节省了大量的io操作 。

缺点：

  • 插入速度严重依赖于插入顺序 ，按照主键的顺序插入是最快的方式，否则将会出现页分裂，严重影响性能。因此，对于InnoDB表，我们一般都会定义一个自增的ID列为主键
  • 更新主键的代价很高 ，因为将会导致被更新的行移动。因此，对于InnoDB表，我们一般定义主键为不可更新
  • 二级索引访问需要两次索引查找 ，第一次找到主键值，第二次根据主键值找到行数据

非聚簇索引：没有其他数据，也没有隐藏列

小结：聚簇索引与非聚簇索引的原理不同，在使用上也有一些区别：

  1. 聚簇索引的叶子节点存储的就是我们的数据记录, 非聚簇索引的叶子节点存储的是数据位置。非聚簇索引不会影响数据表的物理存储顺序。
  2. 一个表只能有一个聚簇索引，因为只能有一种排序存储的方式，但可以有多个非聚簇索引，也就是多个索引目录提供数据检索。
  3. 使用聚簇索引的时候，数据的查询效率高，但如果对数据进行插入，删除，更新等操作，效率会比非聚簇索引低。

P119

InnoDB的B+树索引的注意事项

  • 根页面位置万年不变
    • 一开始插的第一条数据，假设存的页面是A，那么之后当这个页满了，要新加另外一个页，这个时候也会同时生成一个目录页。而这个过程并不是页面A依然是数据页是个叶子结点，而是A页复制自己到新开辟的页面B，自己成为了目录页
  • 内节点目录项记录的唯一性
    • 非聚簇索引，内节点目录项中的索引可能不唯一，因此找数据的时候会不知道去哪个分支，因此需要额外记录逐主键的聚簇索引，这样才能判断
  • 一个页面最少两条记录
    • 不然就是一个链表，二分查找失去了意义

为什么InnoDB数据库不建议主键过长

  • 会使得非聚簇索引存的主键，整体过于庞大

为什么InnoDB数据库建议使用递增的主键（例如自增）

  • 这样使得插入变得高效，不然会调整整体的索引结果

P120

理论上 Hash 索引的效率更高O(1)，为什么不用 Hash 索引？

  • Hash对于范围索引查询，会退化到O(n)
  • Hash底层存储的是无序的，因此如果涉及到 ORDER BY 等查询语句，要另外重新排序
  • 联合索引时，可能多个组合都能得到同一个值，无法对单独一个键来查询
  • 索引列的重复值较多时，查询较慢（一定Hash冲突，链表的O(n)复杂度查询），例如性别列

因此 Hash 索引 常用于等值判断

  • Redis，Memory引擎

InnoDB 有自适应Hash索引

  • 热点数据查询多次以后，将数据页地址放到Hash表中，之后直接跳转，不走B+树

数据结构的演变

  • 二叉搜索树
  • AVL树 / 平衡二叉树
  • B-Tree（M叉树）
  • B+Tree

B-Tree 和 B+Tree 的区别

  • 叶子结点数量和关键字数量关系

    • B-Tree：孩子数量 = 关键字数 +1
    • B+Tree：孩子数量 = 关键字数

  • 内节点的关键字出现位置

    • B-Tree：不会出现在子节点上
    • B+Tree：子节点和内节点上都会出现，并且内子节点关键字一定是对应子节点中的最大或最小

  • 内节点是否保存数据

    • B-Tree：数据 + 索引
    • B+Tree：只有索引

  • 查询流程：

    • B-Tree：会在叶子结点和内节点反复横跳
    • B+Tree：因为关键字都在叶子结点上，且叶子结点构成有序表，所以只在叶子结点上走

B+Tree的优点

  • 查询更稳定
    • 查询次数一定是B+Tree的层数
  • 查询效率更高
    • 内节点只存放索引关键字，没有数据，所以能存放更多关键字，因此分叉更多，整体更矮胖，效率也更高
  • 范围查询的效率更高
    • 因为关键字都在叶子结点上，叶子结点双向有指针指引，加上顺序，范围查询很方便。不会像 B-Tree 一样要中序遍历，在叶子结点和内节点之间反复横跳

为什么一般1-3次IO就能查到数据了

  • 一个数据页16KB，一条索引数据假设是主键是BIGINT类型8字节，地址指针一般也是4-8字节，满打满算一条索引数据占16B，那么一个数据页能存1000条索引，4层就是10^12条索引
  • 因为根页面位置万年不变，所以其实3次IO就对应4层B+Tree，那么已经能覆盖10^12条数据，足以应对大部分的场景

来源：labuladong的算法小抄

1081. Smallest Subsequence of Distinct Characters

  • 思路没啥问题：3个要素
    • 去重：用 boolean[] exists 来记录，如果已经出现，直接跳过
    • 保持原来相对顺序：用单调栈来存（所有这种相对顺序不变的题，都是单调栈）
    • 字典序最小：关于单调栈是否能压调栈顶数来达到字典序更小，额外用一个数组记录之后还有没有栈顶字母，有的话就压掉
  • 循环里的顺序注意：
    • 先判断栈里是否已经有当前元素，有的话是万万不能进入压扁栈顶数的循环的
      • 错误案例：”abcacb”，输出：”acb”
