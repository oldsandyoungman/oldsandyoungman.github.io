P132

限制索引数量

单表索引数量不超过6个，原因：

  • 索引也要占用空间
  • INSERT, DELETE, UPDATE 也要同时对索引进行调整，降低效率
  • 优化器在选择最好的执行计划的时候，要对每个索引进行评判，这里面也有时间损耗

P133

不需要创建索引的地方

  • WHERE / GROUP BY / ORDER BY 没有用到的字段，不需要加索引
  • 数据量小的时候，不需要加索引
    • 例如 < 1000
  • 有大量重复数据的列，不需要加索引
    • 重复度高于10%，就不需要加索引了
  • 经常更新的表，索引不要过多
    • 字段如果经常更新，就别建索引了
    • 表经常更新，索引尽量少
  • 无序的值尽量不要作为索引
    • 因为会经常性，中间插值，页分裂，效率很低
  • 删除不用或少用的索引
  • 冗余索引和重复索引删除
    • 冗余：有联合索引并且最终匹配，单列索引就没有意义了
    • 重复：主键索引上，又另外申明是UNIQUE的唯一索引

P134

服务器优化步骤

  • 观察服务器是否存在周期性波动
    • 是的话，加缓存，或者更改缓存失效策略
  • 观察是否有不规则延迟或卡顿
    • 是的话，开启慢查询
  • 慢查询出结果后，EXPLAIN / SHOW PROFILE
    • 如果是等待时间太长
      • 调优服务器的参数
    • 如果是执行时间太长
      • 索引设计优化
      • JOIN表过多，需要优化
      • 数据表设计优化
  • 如果上述两个策略也没有太大效果，即查询优化到瓶颈了
    • 读写分离（主从架构）
    • 分库分表（垂直分库，垂直分表，水平分表）

统计SQL的查询成本的指令

  • last_query_cost

影响SQL查询效率的两个因素

SQL查询时一个动态的过程，从页加载的角度来看，我们可以得到以下两点结论：

  1. 位置决定效率。如果页就在数据库 缓冲池 中，那么效率是最高的，否则还需要从 内存 或者 磁盘 中进行读取，当然针对单个页的读取来说，如果页存在于内存中，会比在磁盘中读取效率高很多。
  2. 批量决定效率。如果我们从磁盘中对单一页进行随机读，那么效率是很低的(差不多10ms)，而采用顺序读取的方式，批量对页进行读取，平均一页的读取效率就会提升很多，甚至要快于单个页面在内存中的随机读取。

所以说，遇到I/O并不用担心，方法找对了，效率还是很高的。我们首先要考虑数据存放的位置，如果是进程使用的数据就要尽量放到缓冲池中，其次我们可以充分利用磁盘的吞吐能力，一次性批量读取数据，这样单个页的读取效率也就得到了提升

EXPLAIN 的常用字段 type 的优先级

性能的优劣比较：

system > const > eq_ref > ref > fulltext > ref_or_null > index_merge > unique_subquery > index_subquery > range > index > ALL

标准：至少要range以上，range甚至都不推荐
