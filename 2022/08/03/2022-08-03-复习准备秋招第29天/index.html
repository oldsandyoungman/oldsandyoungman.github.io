P125

MyISAM 是可以没有主键的

InnoDB 一定有主键

  • 自己定义了主键，那就是主键了
  • 没有定义的话，如果有 UNIQUE 字段，那就拿来当主键
  • 如果连 UNIQUE 字段也没有，那么自动生成一个 ROW_ID 隐藏列，作为主键

COMPACT 行格式描述

  • 变长字段的长度列表
  • NULL值列表
  • 记录头信息（5字节）
    • delete_mask
    • min_rec_mask
    • record_type
    • heap_no
    • n_owned
    • next_record
  • 真实数据
    • 隐藏列
      • RowID（非必要，没有主键才会自动生成）
      • TransactionID
      • Roll Pointer
    • 显式数据

P126

Dynamic 和 COMPACT 格式的区别

  • 行溢出的处理方式

    （行溢出：如果某个字段直接大于表容量，eg. VARCHAR 65533 = 65533B > 16KB = 16384 B）

    • COMPACT：把多出来的数据，放到另一个地址，本部分留20个字节来记录地址
    • Dynamic：完全行溢出的处理策略，本地不存放数据，只留20个字节来记录地址，存到另一个BLOB位置上

Compressed 相较于Dynamic 格式的区别

  • 额外用 zlib 进行了数据压缩

Redundant 和 COMPACT 格式的区别

  • 字段长度偏移列表 替换了 变长字段的长度列表
    • 没有“变长”：即记录了所有字段的长度，有冗余
    • 多了“偏移”：记录的不是长度，而是相对于首地址的偏移量，相邻地址相减才是长度
  • 没有了 NULL 值列表
  • 记录头信息变化
    • 多了两个字段：n_field和1byte_offs_flag
    • 少了一个字段：没record_type

P127

为什么要有区

  • 如果页在物理上不连续，那么光是磁盘寻道的时间就会远大于读取时间，那么从磁盘读数据到内存效率就会很低
  • 引入区以后，保证一个区里面64个区的存储是连续的，这样就能大大提高从磁盘读取数据的效率

为什么要有段

  • 一个区如果混杂叶子结点和非叶子结点，那么范围查询的时候效率就会极低，所以要给区分类，分出来的结果集合就叫段，即数据段（叶子结点段，即里面都是叶子结点区）、索引段（非叶子结点段，即里面都是非叶子结点区）
  • 段是逻辑的概念，例如叶子结点区的结合，里面有很多个区，但是物理上不一定连续（段除了完整的区，可能还带有零散的页）

为什么要有碎片区

  • 一个段默认是1M，如果一开始数据比较少，正常分配就是索引段占1M，数据段1M，很浪费空间
  • 设定碎片区，即里面的页不属于任何一个段（直属于表空间），而是所有段都可以共用，当某个段的页面超过32个页，之后就会申请完整的区来存储数据

区的分类

根据区的归属，状态，分为2种4类

  • 独立属于表空间的

    • 空闲的区

    • 有剩余空间的碎片区

    • 没有剩余空间的碎片区

  • 附属于某个段的区

    • 附属于某个段的区

独立表空间默认创建

  • 96K = 6* 16KB，即6个页的数据

数据字典

  • 除了用户自定义的数据，为了记录其他的各种信息，例如

    • 这个表在哪个表空间，表有多少列
    • 表的每个字段啥类型

    这些非用户数据的额外数据，也被称为元数据，InnoDB 用一些列的内部系统表来存储，被称为数据词典

  • 它们都是以 B+ 树的形式保存在系统表空间的某个页面中。其中 SYS_TABLES、SYS_COLUMNS、SYS_INDEXES、SYS_FIELDS 这四个表尤其重要，称之为基本系统表 (basic system tables)

  • 用户不能直接访问 InnoDB 的这些内部系统表，除非你直接去解析系统表空间对应文件系统上的文件。不过考虑到查看这些表的内容可能有助于大家分析问题，所以在系统数据库 information_schema 中提供了一些以 innodb_sys 开头的表（相当于SYS_表的副本，内容也不完全一样）

P128

为什么要有主键索引

  • 因为数据一般只存一份（索引可以存多分），聚簇索引最后的叶子结点是要存数据的，所以两个主键索引就浪费了一倍的空间

P129

联合索引的使用方法

  • 最左前缀原则

全文索引的注意事项

  1. 使用全文索引前，搞清楚版本支持情况；
  2. 全文索引比 like + % 快 N 倍，但是可能存在精度问题；
  3. 如果需要全文索引的是大量数据，建议先添加数据，再创建索引

P130

AUTO_INCREMENT 字段为什么不能删唯一索引

因为 AUTO_INCREMENT 能添加的前提只有两种可能

  • 主键索引
  • 唯一索引

在两者都不满足的情况下，就会报错

联合索引删除其中一个字段会怎样

  • 联合索引会重置成少一个字段的，但不会消失

8.0新特性降序索引怎么理解

  • 8.0之前虽然能显示指定降序，但是底层依然是升序索引，使用的时候反向扫描，效率很差

  • 8.0才是真觉得降序索引

  • 即使是8.0，也要搜索的顺序和建表指定的顺序一致，才真正用到了索引

8.0新特性隐藏索引的用途

  • 试探能不能删索引
    • 直接删可能会报错中，之后加回去会浪费大量时间
    • 先设成隐藏再删除，更保险
  • 试探索引有没有用
    • 有些索引设了但是并没有加快效率，不如删了
    • 先设成隐藏，测试效率如果没有降低，可以删除这个索引
  • （注意，主键不能设成隐藏索引）
  • 索引隐藏，内容依然和正常索引一样实时更新，所以长期隐藏的建议删除

P131

索引的创建原则

  • 唯一字段，建议加索引（Alibaba手册写的是一定）
    • 查到就会立即停止，而不会继续找左右边界
    • 虽然加了索引后，插入数据会每次多插内容多消耗时间，但查询带来的提升远大于这个，前者可忽略
  • 经常 WHERE 查询的字段
    • 索引会排序嘛，二分查找肯定能提升效率
    • 即使只是普通索引，也能大大提升效率
  • 经常被 GROUP BY / ORDER BY 的字段
    • 索引已经帮排好序了，不用你再排序了，ORDER BY 的语句肯定能提升性能
    • 同样索引已经帮排好序了，相同的值放在一起，因此再GROUP BY 的时候，不用左找一个，右找一个合并，效率肯定提升
